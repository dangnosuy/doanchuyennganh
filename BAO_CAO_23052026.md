## 1. Tổng quan đồ án

**Tên:** Hệ thống Multi-Agent Reinforcement Learning hỗ trợ kiểm thử xâm nhập tự động

**Mục tiêu:** Phát hiện và khai thác tự động hai nhóm lỗ hổng phổ biến:
- `BAC` — Broken Access Control (leo thang quyền, IDOR)
- `BLF` — Business Logic Flaw (lỗi luồng nghiệp vụ: giảm giá, âm số lượng, bypass checkout)

**Tiếp cận:** Hệ thống nhiều agent AI phối hợp theo mô hình **Orchestrator + Sub-agents**, mỗi agent đóng một vai trò chuyên biệt trong pipeline pentest.

### 3.2 Vấn đề #2 — Red viết "script spec", không phải "chiến lược"

**Trước (SHOT PLAN):**
```text
=== EXECUTION SHOT PLAN ===
Shot 1 - single-script:
  Goal: Khai thác IDOR trên /api/Users/{id}
  Actions:
    1. Login as user A, lấy token
    2. Dùng token A để GET /api/Users/2
    3. Nếu status 200 → IDOR confirmed
  Must save: baseline.req.txt, probe.req.txt, result.json
=== END EXECUTION SHOT PLAN ===
```

**Vấn đề:** Red đang viết script specification, không phải chiến lược. Exec nhận được này và phải gen 1 Python script hoàn chỉnh → dễ syntax error, thiếu error handling, không handle SPA timing.

**Sau (EXECUTION GUIDE):**
```text
=== EXECUTION GUIDE ===
Approach: api_first
Auth setup: cần JWT bearer — login trước nếu chưa có
Steps:
  1. [Baseline] GET /api/Users/1 với token user A → expect 200, ghi lại user_id/email
  2. [Probe]    GET /api/Users/2 với token user A → check status + body
  3. [Verify]   Nếu body chứa email/data của user khác → IDOR confirmed
Fallback nếu step fail:
  - Nếu /api/Users trả 404: thử /rest/user/whoami để confirm auth
  - Nếu 403: endpoint có ACL → thử method bypass (PUT thay GET)
Success condition: User A đọc được dữ liệu cụ thể của User B
=== END EXECUTION GUIDE ===
```

**Lợi ích:** Red tập trung vào chiến lược. Exec tự quyết định dùng tool nào (curl, browser, fetch) tùy tình huống.

### 3.3 Vấn đề #3 — Exec "bắn một phát" không đủ cho bug phức tạp

**Trước:** Exec nhận SHOT PLAN → gen 1 Python script → chạy 1 lần → báo kết quả.

Vấn đề với BLF (stateful bugs):
- Cart → Checkout → Payment là multi-step workflow
- 1 script không handle được DOM timing, redirect, session state
- Script fail → Exec báo FAILED → bug bị bỏ qua

**Sau (Tool-loop adaptive):**

```
Exec nhận EXECUTION GUIDE
  Round 1: curl GET /api/BasketItems/1 → baseline qty=1
  Round 2: curl PUT /api/BasketItems/1 {"quantity": -1} → status 200?
  Round 3: curl GET /api/BasketItems/1 → verify qty=-1 → total âm?
  Round 4: Browser snapshot → confirm giỏ hàng âm tiền
  Round 5: Tổng hợp → EXPLOITED (before: $5.99, after: -$5.99)
```

- Tối đa 15 rounds tool calls
- Chọn tool phù hợp từng bước (curl cho API, browser cho SPA UI)
- Chỉ gen Python PoC script SAU KHI xác nhận EXPLOITED

## 6. Ý nghĩa kiến trúc — liên hệ lý thuyết MARL

### Phân vai agent theo lý thuyết:

| Agent | Vai trò lý thuyết | Đặc điểm |
|---|---|---|
| ManageAgent | **Orchestrator** (Central Coordinator) | Nắm toàn bộ state, điều phối sub-agents |
| CrawlAgent | **Environment Observer** | Thu thập observation từ target |
| VulnHunterAgent | **Hypothesis Generator** | Sinh action candidates |
| RedTeam | **Strategy Proposer** | Đề xuất chiến lược cụ thể |
| BlueTeam | **Critic / Adversarial Reviewer** | Phản biện để tăng chất lượng |
| ExecAgent | **Executor / Actor** | Thực thi action trong environment |

### Cơ chế phối hợp:

- **Red-Blue Debate**: Không phải dân chủ biểu quyết, mà là **quality gate**. Blue reject nếu chiến lược không có fallback path hoặc verify condition mơ hồ.
- **Context Isolation**: Mỗi agent chỉ nhận context cần thiết (không cần toàn bộ conversation history) → tiết kiệm token, giảm overfitting.
- **Adaptive Execution**: ExecAgent tự quyết định tool nào dùng ở mỗi round → không bị ràng buộc bởi script tĩnh.
