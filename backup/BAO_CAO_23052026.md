# Báo cáo Tiến độ Đồ án — MARL Pentest Agent
**Ngày:** 23/05/2026 | **Thời gian trình bày:** ~30 phút (bao gồm demo)

---

## 1. Tổng quan đồ án

**Tên đề tài:** Hệ thống Multi-Agent Debate For Business Logic Vulnerability Detection

**Mục tiêu:** Xây dựng một hệ thống nhiều agent AI có khả năng tự động phát hiện và khai thác hai nhóm lỗ hổng bảo mật web phổ biến:

- **BAC (Broken Access Control)** — Lỗ hổng kiểm soát truy cập bị phá vỡ: bao gồm leo thang quyền hạn, IDOR (Insecure Direct Object Reference — truy cập tài nguyên người dùng khác thông qua ID dự đoán được).
- **BLF (Business Logic Flaw)** — Lỗi luồng nghiệp vụ: các lỗ hổng xảy ra do logic ứng dụng không được kiểm tra đúng đắn, ví dụ: đặt số lượng âm trong giỏ hàng, bỏ qua bước xác thực, thao tác giá trị phía client.

**Phạm vi áp dụng:** Hệ thống được thiết kế để hoạt động trên **các ứng dụng web thực tế nói chung**, không giới hạn vào một lab hay nền tảng cụ thể nào.

**Tiếp cận:** Sử dụng mô hình **Orchestrator + Sub-agents** — một agent điều phối trung tâm (ManageAgent) quản lý vòng đời toàn bộ pipeline, trong khi các agent chuyên biệt đảm nhiệm từng giai đoạn của quá trình kiểm thử bảo mật (pentest).

---

## 2. Kiến trúc tổng thể hiện tại

```
INPUT: "Test https://target.com user:admin pass:secret"
         │
         ▼
      CrawlAgent          ← Thu thập thông tin mục tiêu (RECON)
          │
          ▼
      VulnHunter          ← Phân tích và sinh giả thuyết lỗ hổng
          │
          ▼
┌─────────────────────────────────────────────────────┐
│              ManageAgent (ORCHESTRATOR)             │
│                                                     │
│  Phase 0: Context Review (đánh giá trước dispatch) │
│    ├─ Đánh giá chất lượng recon (endpoint count)   │
│    ├─ Kiểm tra trạng thái xác thực (auth status)   │
│    ├─ Phân loại bug: ẩn danh / yêu cầu auth        │
│    └─ Ra quyết định: proceed / exec_login_first     │
│                                                     │
│  Bug Loop: anonymous bugs trước → auth bugs sau     │
│    ├─ Auth recovery nếu thiếu session               │
│    └─ Preflight check trước khi dispatch            │
└──────────────┬──────────────────────────────────────┘
               │
        ┌──────┴─────────────┐
        ▼                    ▼
 [RedTeam ↔ BlueTeam]   ExecAgent
   (Debate)               (Execute)
```

### Vai trò từng agent:

| Agent | Vai trò | Input | Output |
|---|---|---|---|
| **CrawlAgent** | Thu thập thông tin mục tiêu (Reconnaissance) | URL + credentials | recon.md, crawl_raw.json |
| **VulnHunterAgent** | Sinh giả thuyết lỗ hổng từ dữ liệu recon | recon.md + HTTP evidence | risk-bug.json (8–15 bugs) |
| **RedTeam** | Đề xuất chiến lược khai thác cho từng bug | Bug dossier | EXECUTION GUIDE |
| **BlueTeam** | Phản biện và kiểm tra chất lượng chiến lược | Red's guide | APPROVE / REJECT |
| **ExecAgent** | Thực thi khai thác thực tế trên target | Approved guide + auth session | EXPLOITED / PARTIAL / FAILED |
| **ManageAgent** | Điều phối toàn bộ pipeline và tổng hợp báo cáo | Tất cả agent output | Final report |

## 3. Vấn đề phát sinh và giải pháp

Trong quá trình kiểm thử pipeline trên các web application thực tế, chúng tôi xác định được 5 vấn đề cốt lõi cần giải quyết để pipeline hoạt động đáng tin cậy.

---

### 3.1 Vấn đề #1 — Login thất bại do giao diện bị che khuất (Overlay Problem)

**Hiện tượng quan sát:**

Nhiều ứng dụng web hiện đại hiển thị các lớp phủ giao diện (overlay) khi người dùng vào trang lần đầu: popup chào mừng, banner cookie consent (thông báo chấp thuận cookie theo GDPR), dialog thông báo chính sách. Các overlay này che khuất các thành phần tương tác phía dưới, khiến Playwright — công cụ tự động hóa trình duyệt — không thể click vào nút Login.

**Hậu quả:** Toàn bộ pipeline mất `authenticated context` (phiên đăng nhập), dẫn đến phần lớn bug yêu cầu xác thực bị đánh dấu `BLOCKED_AUTH`.

**Giải pháp — Hàm `_dismiss_overlays(page)`:**

Trước khi tương tác với form đăng nhập, CrawlAgent tự động dismiss (đóng/bỏ qua) các overlay phổ biến theo danh sách selector ưu tiên. Hàm được gọi **hai lần**: trước khi navigate đến trang đăng nhập và sau khi click vào link/nút dẫn đến form login — để xử lý trường hợp overlay xuất hiện lại sau khi điều hướng trang.

```python
# crawl_agent.py — _dismiss_overlays(page)
# Duyệt qua danh sách CSS selector của các overlay phổ biến:
# - Nút đóng modal (button.close, [aria-label="Close"], [aria-label="Dismiss"])
# - Backdrop của các thư viện UI phổ biến (Angular CDK, Bootstrap modal)
# - Nút chấp nhận cookie consent (GDPR): "Accept", "Got it", "I agree", "Accept all"
# - Banner chào mừng của ứng dụng
# Với mỗi selector: kiểm tra visible + enabled → click nếu thỏa → đợi 250ms
# Nếu đóng được ≥1 overlay → ghi log số lượng đã dismiss
```

**Phạm vi:** Hàm này được thiết kế để hoạt động với các overlay phổ biến trên web nói chung, không phụ thuộc vào framework hay nền tảng cụ thể.

### 3.2 Vấn đề #2 — Tài khoản chưa tồn tại, không thể đăng nhập

**Hiện tượng quan sát:**

Khi thực hiện pentest trên môi trường lab hoặc staging, tài khoản test được cung cấp có thể chưa tồn tại trên hệ thống (REST API login trả về HTTP 401 Unauthorized cho mọi endpoint). Trước đây, pipeline sẽ dừng lại và báo lỗi.

**Giải pháp — Hàm `_auto_register(target_url, credentials)`:**

Khi REST API login thất bại với mọi endpoint đã thử, CrawlAgent tự động thực hiện đăng ký tài khoản mới với credentials được cung cấp, sau đó thử đăng nhập lại.

```python
# crawl_agent.py — _auto_register(target_url, credentials)
# Thử tạo tài khoản qua các registration endpoint phổ biến:
#   /api/Users, /rest/user/register, /api/register, /register,
#   /api/auth/register, /api/signup, /signup
# Với mỗi endpoint, thử 3 variants body:
#   {"email": ..., "password": ...}
#   {"username": ..., "password": ...}
#   {"email": ..., "password": ..., "passwordRepeat": ...}
# Nếu server trả HTTP 200/201 → đăng ký thành công → retry login
```

**Điều kiện kích hoạt:** Chỉ thực hiện **một lần** (cờ `_auto_register_attempted`) để tránh vòng lặp vô hạn. Nếu auto-register cũng thất bại, pipeline tiếp tục ở chế độ anonymous (không xác thực).

### 3.3 Vấn đề #3 — Red Team viết "script specification" thay vì "chiến lược"

**Vấn đề với định dạng SHOT PLAN (cũ):**

```text
=== EXECUTION SHOT PLAN ===
Shot 1 - single-script:
  Goal: Khai thác IDOR trên /api/Users/{id}
  Actions:
    1. Login as user A, lấy token
    2. Dùng token A để GET /api/Users/2
    3. Nếu status 200 → IDOR confirmed
  Must save: baseline.req.txt, probe.req.txt, result.json
=== END EXECUTION SHOT PLAN ===
```

Red Team đang viết **specification cho Python script** (liệt kê từng bước cụ thể theo code flow), không phải chiến lược tấn công. ExecAgent nhận được đây và phải sinh ra một Python script hoàn chỉnh → dễ xảy ra lỗi cú pháp, thiếu error handling, không xử lý được các tình huống bất ngờ của ứng dụng.

**Giải pháp — Định dạng EXECUTION GUIDE (mới):**

```text
=== EXECUTION GUIDE ===
Approach: api_first
Auth setup: cần JWT bearer — login trước nếu chưa có
Steps:
  1. [Baseline] GET /api/Users/1 với token user A → expect 200, ghi lại user_id/email
  2. [Probe]    GET /api/Users/2 với token user A → check status + body
  3. [Verify]   Nếu body chứa email/data của user khác → IDOR confirmed
Fallback nếu step fail:
  - Nếu endpoint trả 404: thử endpoint tương tự để xác nhận auth hoạt động
  - Nếu 403: endpoint có ACL → thử method bypass (PUT thay GET)
Success condition: User A đọc được dữ liệu cụ thể của User B
=== END EXECUTION GUIDE ===
```

**Phân biệt rõ vai trò:**
- **Red Team** chịu trách nhiệm **chiến lược** (approach, fallback, success condition).
- **ExecAgent** chịu trách nhiệm **quyết định tool** (dùng curl, fetch, browser, hay Python script) phù hợp với từng tình huống cụ thể tại runtime.

**Lợi ích so với cũ:** Red không cần biết target dùng framework gì, API trả về JSON hay HTML, hay phiên đăng nhập đang dùng cookie hay JWT. ExecAgent tự thích nghi.

### 3.4 Vấn đề #4 — ExecAgent "bắn một phát" không đủ cho bug phức tạp

**Trước:** ExecAgent nhận SHOT PLAN → sinh 1 Python script → chạy 1 lần → báo kết quả. Với các lỗ hổng stateful (nhiều trạng thái, nhiều bước), một script duy nhất không đủ để xử lý:

- **Timing phụ thuộc vào ứng dụng** (ví dụ: cần chờ trang load xong trước khi đọc DOM).
- **Session state**: mỗi request trong workflow phụ thuộc kết quả của request trước.
- **Redirect chains**: ứng dụng redirect nhiều lần trước khi đến endpoint đích.

Khi script fail → ExecAgent báo FAILED → bug bị bỏ qua, dù lỗ hổng thực sự tồn tại.

**Sau — Tool-loop adaptive (hàm `_exploit_via_tools`):**

ExecAgent chạy theo vòng lặp tool calls, mỗi round sử dụng một tool phù hợp với mục tiêu của bước đó:

```
ExecAgent nhận EXECUTION GUIDE
  Round 1: [Baseline] Gửi request lấy giá trị ban đầu → ghi lại
  Round 2: [Probe]    Gửi request khai thác → quan sát response
  Round 3: [Verify]   So sánh trạng thái trước/sau → xác nhận khai thác
  Round 4: [Snapshot] Chụp bằng chứng (response body, screenshot nếu cần)
  Round 5: [Summary]  Tổng hợp → tuyên bố EXPLOITED / PARTIAL / FAILED
```

**Cơ chế chọn tool (hàm `_choose_exploit_mode`):**

| Loại workflow | Mode được chọn |
|---|---|
| `Approach: browser_first` hoặc `mixed` trong GUIDE | `tool_loop` |
| Bug stateful (cart, checkout, balance, transfer...) | `tool_loop` |
| EXECUTION GUIDE format (mới) | `tool_loop` (mặc định) |
| `Approach: api_first` đơn giản | `script_first` (legacy) |

**Giới hạn:** Tối đa 15 rounds tool calls để kiểm soát chi phí và tránh vòng lặp vô hạn.

**PoC script chỉ được sinh sau khi xác nhận EXPLOITED:** Hàm `_generate_poc_from_evidence` chỉ được gọi khi exploit log chứa `FINAL: EXPLOITED` — tránh sinh script vô nghĩa cho các lần thử thất bại.

### 3.5 Vấn đề #5 — ManageAgent không có "bức tranh tổng thể" trước khi dispatch

**Trước:** ManageAgent dispatch từng bug tuần tự mà không biết:
- Recon có đủ thông tin không (hay chỉ crawl được homepage)?
- Đã có authenticated session chưa (hay cần login trước)?
- Nên ưu tiên bug nào trước để tận dụng context từ bug trước?

Hậu quả: nhiều bug auth-required bị dispatch trước khi có session → BLOCKED_AUTH hàng loạt.

**Sau — Phase 0: Context Review (hàm `_phase0_context_review`):**

Trước khi dispatch bất kỳ bug nào, ManageAgent thực hiện một lượt đánh giá toàn diện:

```python
def _phase0_context_review(self) -> dict:
    # 1. Đánh giá chất lượng recon:
    #    "rich"    → ≥10 endpoint phát hiện được
    #    "minimal" → 5–9 endpoint
    #    "poor"    → <5 endpoint, không có HTTP examples
    #
    # 2. Đánh giá trạng thái xác thực (auth status):
    #    "verified" → có session được xác nhận bởi auth-checking endpoint
    #    "partial"  → có session nhưng chưa verify được
    #    "none"     → không có session nào
    #
    # 3. Phân loại bugs:
    #    anonymous_bugs  → không cần đăng nhập
    #    auth_bugs       → cần đăng nhập
    #    bugs_with_examples → có HTTP evidence từ crawl
    #
    # 4. Đưa ra khuyến nghị:
    #    "proceed"           → tất cả OK, chạy ngay
    #    "exec_login_first"  → auth_bugs > anon_bugs và chưa có session
    #    "proceed_cautious"  → recon kém, chạy nhưng kỳ vọng thấp
```

**Sắp xếp thứ tự ưu tiên (hàm `_sort_bugs_by_auth_priority`):**

```python
def _sort_bugs_by_auth_priority(self):
    # Nhóm 1 — Anonymous bugs (không cần auth):
    #   CRITICAL → HIGH → MEDIUM → LOW
    # Nhóm 2 — Auth-required bugs:
    #   CRITICAL → HIGH → MEDIUM → LOW
    # Lý do: chạy anonymous bugs trước có thể:
    #   - Thu thập thêm thông tin về cấu trúc ứng dụng
    #   - Khai thác bug anonymous để có thêm context
    #   - Tạo cơ hội build auth từ các endpoint public
```

**Auth recovery thay vì block (hàm `_ensure_auth_or_skip`):**

Khi gặp auth-required bug mà chưa có session, thay vì block ngay:
1. Lấy thông tin `login_discovery` từ `auth_context.json` (CrawlAgent đã ghi lại endpoint đăng nhập thành công, cơ chế xác thực, body fields).
2. Giao ExecAgent thực hiện đăng nhập thực tế.
3. Nếu đăng nhập thành công → cập nhật `has_authenticated_context` → tiếp tục với bug.
4. Chỉ `BLOCKED_AUTH` nếu cả hai bước trên thất bại và bug không có HTTP examples để bám theo.

### 3.6 Vấn đề #6 — VulnHunterAgent không có bằng chứng HTTP thực tế

**Trước:** VulnHunterAgent chỉ đọc `recon.md` (file text tổng hợp). Không có dữ liệu request/response cụ thể → LLM sinh bug candidates mang tính suy luận, thiếu HTTP evidence để ExecAgent bám theo khi khai thác.

**Sau — Inject HTTP evidence từ crawl data:**

**Hàm `_extract_raw_endpoints(anon_data, auth_sessions)`** (trong CrawlAgent):

Sau khi crawl xong, CrawlAgent trích xuất và lưu vào `crawl_raw.json["raw_endpoints"]` danh sách các HTTP request/response mẫu cho mỗi endpoint duy nhất (theo phương thức + path):

```python
# Mỗi entry trong raw_endpoints chứa:
{
  "method": "GET",
  "path": "/api/Users/1",
  "status": 200,
  "request": {
    "headers": {"Authorization": "Bearer ..."},
    "body": null
  },
  "response": {
    "body_snippet": '{"id":1,"email":"admin@...","role":"admin"}',
    "headers": {"Content-Type": "application/json"}
  },
  "auth_session": "authenticated"   # hoặc "anonymous"
}
```

Ưu tiên lưu: entry có status 2xx được ưu tiên hơn 4xx; session xác thực được ưu tiên hơn anonymous. Bỏ qua static assets (`.js`, `.css`, `.png`...).

**Hàm `_match_raw_endpoints()`** (trong VulnHunterAgent):

Khi LLM sinh bug candidate nhưng không cung cấp `http_examples`:
- Tìm kiếm trong `raw_endpoints` entry có method + path khớp với endpoint của bug.
- Tự động inject HTTP request/response example thực tế vào field `http_examples` của bug.
- ExecAgent sau đó có thể đọc `http_examples` để biết chính xác format header, body, và response trông như thế nào.

## 5. So sánh pipeline cũ và mới

```
                 PIPELINE CŨ (Shot-based)
─────────────────────────────────────────────────────
Input
  │
  ▼
CrawlAgent
  ├─ Nếu overlay che form login → Playwright fail → mất auth
  └─ Không lưu HTTP examples → VulnHunter chỉ đọc text
  │
  ▼
VulnHunterAgent
  └─ Chỉ đọc recon.md → bug candidates thiếu HTTP evidence
  │
  ▼
[Per-bug loop — không nhìn bức tranh tổng]:
  Manager: dispatch tuần tự, không biết auth status
  Red  → viết SHOT PLAN (script format, chi tiết cứng nhắc)
  Blue → review "shot plan có đúng format không?"
  Exec → sinh 1 Python script → chạy 1 lần
       → Fail thì báo FAILED (không retry thông minh)
       → Luôn sinh PoC dù chưa xác nhận khai thác
  Manager → block ngay nếu thiếu auth (BLOCKED_AUTH)
─────────────────────────────────────────────────────


                PIPELINE MỚI (Orchestrator-driven)
─────────────────────────────────────────────────────
Input
  │
  ▼
CrawlAgent
  ├─ Dismiss overlays trước khi tương tác form login
  ├─ Auto-register nếu tài khoản chưa tồn tại → retry login
  ├─ Log rõ Anonymous vs Authenticated crawl session
  └─ Lưu raw HTTP examples → crawl_raw.json

  │
  ▼
VulnHunterAgent
  ├─ Đọc recon.md
  ├─ Inject http_examples từ crawl_raw.json
  └─ Lọc bỏ bug candidates chỉ dựa trên metadata ứng dụng

  │
  ▼
ManageAgent — Phase 0: Context Review
  ├─ Đánh giá recon quality (rich / minimal / poor)
  ├─ Đánh giá auth status (verified / partial / none)
  ├─ Nếu thiếu auth và nhiều auth bugs → Exec login trước
  └─ Sort bugs: anonymous → auth-required, theo risk level

  │
  ▼
[Per-bug loop — Orchestrator-driven]:
  Red  → viết EXECUTION GUIDE (approach + fallback + verify condition)
  Blue → review chất lượng guide (có fallback? verify condition rõ?)
  Exec → tool-loop adaptive (tối đa 15 rounds)
       ├─ api_first: fetch/curl cho REST API endpoints
       ├─ browser_first: Playwright cho UI phức tạp
       └─ mixed: kết hợp cả hai tùy bước
  → EXPLOITED → sinh Python PoC reproduce
  → Auth fail mid-bug → Manager thử auth recovery trước khi block
```

## 6. Ý nghĩa kiến trúc — liên hệ lý thuyết MARL

### Phân vai agent theo lý thuyết MARL (Multi-Agent Reinforcement Learning):

| Agent | Vai trò lý thuyết | Đặc điểm trong hệ thống |
|---|---|---|
| ManageAgent | **Central Coordinator (Orchestrator)** | Nắm toàn bộ state, điều phối sub-agents, ra quyết định routing |
| CrawlAgent | **Environment Observer** | Thu thập observation từ target environment (HTTP traffic) |
| VulnHunterAgent | **Hypothesis Generator** | Sinh tập action candidates (bug hypotheses) từ observation |
| RedTeam | **Strategy Proposer** | Đề xuất exploit strategy cụ thể cho từng bug candidate |
| BlueTeam | **Critic / Adversarial Reviewer** | Phản biện để loại bỏ chiến lược kém chất lượng trước khi thực thi |
| ExecAgent | **Actor / Executor** | Thực thi action trong environment và trả về reward signal (EXPLOITED/FAILED) |

### Cơ chế phối hợp quan trọng:

**Red-Blue Debate — Quality Gate, không phải vote:**
- BlueTeam không "biểu quyết đồng thuận" — đây là một **cửa kiểm soát chất lượng**.
- Blue REJECT nếu chiến lược không có `fallback path` rõ ràng, `verify condition` mơ hồ, hoặc không có `success condition` đo được.
- Mục tiêu: ngăn ExecAgent lãng phí tài nguyên vào chiến lược chắc chắn sẽ thất bại.

**Context Isolation — Giảm token consumption và overfitting:**
- Mỗi agent chỉ nhận context **cần thiết** cho nhiệm vụ của mình, không nhận toàn bộ conversation history.
- Ví dụ: ExecAgent nhận `EXECUTION GUIDE + auth context + recon summary` — không cần biết lịch sử debate Red-Blue.
- Lợi ích kép: tiết kiệm token API và giảm nguy cơ agent bị "distracted" bởi thông tin không liên quan.

**Adaptive Execution — Tách chiến lược khỏi implementation:**
- ExecAgent quyết định **tool nào** dùng ở mỗi round dựa trên kết quả thực tế của round trước.
- Tách biệt hoàn toàn "chiến lược tấn công" (Red) với "cách thực hiện" (Exec) → Red không cần biết target là ứng dụng loại gì.
